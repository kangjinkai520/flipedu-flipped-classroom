# AI 驱动核心：RAG 与流式响应 - 演讲稿

## 1. 开场与流式响应 (Slide 1)
大家好，我是康金恺。今天为大家介绍我们项目中最核心的 AI 驱动模块。

首先是**流式对话体验**。大家可以看到，在我们的应用中，AI 的回复不是一次性弹出的，而是像真人打字一样逐字显示。
这背后是因为我们开启了大模型 API 的 `stream=true` 选项。后端在接收到用户的症状描述（如“肚子疼”）后，不会等待 AI 生成完整的长篇大论，而是实时提取 API 返回的 `delta` 增量内容。
我们编写了专门的生成器，将这些片段实时封装成符合 SSE（Server-Sent Events）标准的 JSON 格式，源源不断地推送给前端。前端监听到数据后，立即将其追加到聊天气泡中，从而彻底消除了用户的等待焦虑。

## 2. 对话记忆与全链路通信 (Slide 2)
为了让 AI 像医生一样能“记住”病人的病情，我们需要维护**对话记忆**。
我们采用了一种高效的“滑动窗口”机制：在每次向 API 发送请求时，系统会自动抓取并倒序排列最近的几轮历史记录（如最近 10 条）。这种做法既保证了 AI 能联系上下文（知道你刚才说了“肚子疼”，现在问“吃什么药”是指针对肚子疼的药），又通过限制窗口大小，有效控制了 Token 的消耗，降低了运营成本。
整个**数据通信**链路也非常清晰：前端 `uni.request` 发起请求 -> 后端 `Django` 路由解析 -> 触发 `view.py` 逻辑 -> 建立流式响应通道 -> 实时回传结果。

## 3. RAG 检索增强生成 (Slide 3)
最后是保证医疗建议专业性的关键——**RAG（检索增强生成）实现**。我们设计了**“两阶段检索”**策略来确保准确度：
*   **第一阶段（粗排）**：利用 `gte_base_zh` 向量模型，计算用户问题与知识库文档的余弦相似度，快速从海量数据中捞出最相关的 10 个候选文档。
*   **第二阶段（精排）**：为了解决“字面相似但含义不同”的问题，我们引入了 `bge_reranker_base` 重排模型。它利用 Transformer 的注意力机制，对“候选文档”和“用户问题”进行深度语义比对。
    *   *举个例子*：用户描述“我咳血，心跳很快”，重排模型能精准识别出文档中关于“充血性心力衰竭”或“实性心动过速”的描述，计算出高置信度的相关性分数（0~1之间），最终只将最匹配的专业知识投喂给大模型。

通过这套组合拳，我们实现了既快、又准、又流畅的 AI 医疗咨询体验。谢谢大家！
